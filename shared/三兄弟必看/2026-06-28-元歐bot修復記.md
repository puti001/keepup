# 元歐 Telegram bot (@puti_yuan_bot) 修復記

> 日期：2026-06-28
> 修復者：元歐（小歐協助攻因分析）

---

## 問題

Windows 上的 `opencode serve` 接受 API 但回不了 prompt 結果。
Telegram bot 在群組問話時回應跑到私訊（反之亦然）。

## 根因

### 1. chatIdInstance 是全域 singleton

`event-subscription-service.js` 的 `chatIdInstance` 是 class field（可變變數），
後來的訊息會覆蓋前一個訊息的目標 chat ID。

時間軸：
```
t1: 群組A說「大家好」→ chatIdInstance = 群組A → 送 AI 處理（async）
t2: 師訓說一句話     → chatIdInstance = 師訓   ← 被蓋掉！
t3: t1 的 AI 回覆下來 → 看 chatIdInstance = 師訓 → 送錯地方 ❌
```

### 2. attachManager 不存 chatId

`attach-manager.js` 的 `AttachedSessionState` 沒有 `chatId` 欄位，
attach session 時沒有綁定發起位置的 chat ID。

### 3. 已 attach 的 session 不更新 chatId

`attach-service.js` 中，如果 session 已 attach（`alreadyAttached=true`），
不會再呼叫 `attachManager.attach()`，導致 chatId 停留在第一次 attach 時的值。

---

## 修復（三處 code 改動）

### 檔案 1：`attach-manager.js`

```diff
- attach(sessionId, directory) {
+ attach(sessionId, directory, chatId) {
    this.state = {
      sessionId,
      directory,
      busy: false,
+     chatId,
    };
  }

+ getChatId() {
+   return this.state?.chatId ?? null;
+ }
```

### 檔案 2：`attach-service.js`

```diff
  } else {
    summaryAggregator.setSession(session.id);
    summaryAggregator.setBotAndChatId(bot, chatId);
+   attachManager.attach(session.id, session.directory, chatId);
  }
```

### 檔案 3：`event-subscription-service.js`

將 `chatIdInstance` 改為 getter，優先從 attachManager 取正確值：

```js
get chatIdInstance() {
  const attached = attachManager.getSnapshot();
  if (attached?.chatId) {
    return attached.chatId;
  }
  return this._chatIdInstance;
}
set chatIdInstance(v) {
  this._chatIdInstance = v;
}
```

|------|-----|
| serve port | 4096 |
